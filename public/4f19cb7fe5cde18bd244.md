## はじめに 🤔

GitHub Copilot は、今やコード補完ツールにとどまらない存在になってきています。  
最近、私が日常的に使うようになった **GitHub Copilot app** は、従来の GitHub Copilot CLI とは明らかに異なる体験を提供しています。

この記事では、GitHub Copilot app がどのような位置づけにあるのかを、GitHub Copilot CLI との違いを軸に考察します。技術的な機能比較というよりは、それぞれが「何を目指しているのか」という設計思想の違いを整理するものです。まず GitHub Copilot app そのものを定義するところから始めましょう。

## GitHub Copilot app とは 🌐

GitHub Copilot app は、GitHub Copilot のエコシステムの上に構築されながらも、それを超えた方向性を持つアプリケーションです。  
コードを書く支援だけでなく、**ブランチ作成 → コミット → PR 作成 → レビュー → マージまでのライフサイクル全体をエージェンティック（AI が自律的に複数の操作を連続実行する動作様式）に進めること**を指向しています。

つまり、GitHub Copilot app は単なる「コーディングアシスタント」ではありません。  
GitHub の全体的なワークフローを、AI がひとまとまりの体験として扱えるようにするためのアプリケーションです。

単に Web の GitHub を別 UI で表示するフロントエンドでもありません。  
GitHub Copilot app が目指しているのは、**AI ネイティブな方向性**です。では、この方向性は GitHub Copilot CLI とどのように異なるのでしょうか。

## GitHub Copilot CLI との違い 🔧

GitHub Copilot CLI もまた、近年の進化により PR 作成やブランチ操作など、多くのことが可能になってきています。  
では、GitHub Copilot app との違いは何でしょうか。

私は、両者の差は「機能の有無」よりも、**根底にある設計思想**にあると考えています。

### GitHub Copilot CLI が目指すもの

GitHub Copilot CLI は、完全にコマンドベースのターミナルネイティブなエージェント実行環境を基礎として発展してきたツールです。  
ターミナルやスクリプトとの親和性が高く、**高い制御性**を持っています。

この特性は、熟練した開発者が細かく挙動をコントロールしたい場面や、CI/CD パイプラインの中で AI を呼び出したい場面に非常に向いています。  
言い換えると、GitHub Copilot CLI は「人間が使うツール」としてだけでなく、**AI がさらに AI を呼び出すためのコマンドとしても優れた存在**です。

### GitHub Copilot app が目指すもの

一方の GitHub Copilot app は、**ヒューマンフレンドリーでエージェンティックな GitHub フロントエンド**です。  
GitHub Copilot CLI の高い制御性と引き換えに、取っつきやすさと使いやすさを重視しています。

「ブランチを切って、コードを書いて、コミットして、PR を出して、レビューして、マージする」という一連の流れを、自然な言葉で依頼するだけで進められる体験。それが GitHub Copilot app の方向性です。

コマンドの細かい引数を覚えなくても、フラグを調べなくても、**ただ「こうしたい」と伝えるだけで動く**。この違いは小さいようで、実際の体験としては大きく異なります。

この設計思想の違いをもう少し掘り下げると、「AI ネイティブ」というキーワードで整理できます。

## AI ネイティブなフロントエンドという位置づけ 🤖

「AI ネイティブ」という言葉をあえて使っているのは、従来の GitHub Web UI を AI に対応させたものとは本質的に異なるからです。

従来の発想は「既存の UI に AI の機能を追加する」でした。たとえば、PR 画面にサジェスト機能を追加する、コードに補完機能を差し込む、といった形です。  
しかし GitHub Copilot app が目指しているのは、**そもそも AI を中心に据えた上でワークフロー全体を設計すること**です。

この違いは、利用者の操作体験にも現れます。  
AI ネイティブなフロントエンドは、人間が各操作を個別に実行するのではなく、意図を伝えれば AI が適切な順序でアクションを組み合わせて実行します。GitHub Copilot app は、まさにこの体験を中心に設計されています。

こうした AI ネイティブな設計が、ある具体的な感覚として現れています。それが「離れない」体験です。

## 「離れない」体験設計 💡

私が GitHub Copilot app を使っていて特に感じるのが、**「アプリから離れずにさまざまな作業ができる」** という体験です。

コードを書く、ブランチを切る、PR を作る、差分をレビューする。これらは従来、それぞれ異なる操作場所やコマンドへの移動を伴っていました。ターミナルを開く、GitHub の Web サイトに移動する、PR ページを確認する……こうした文脈の切り替えは、思った以上に集中を妨げます。

GitHub Copilot app はこの「コンテキストスイッチ」を極力なくすことを指向しています。  
ひとつの場所で意図を伝えれば、あとは AI がライフサイクル全体を通じて動いてくれる。これは単なる利便性の向上ではなく、**AI との協働という体験の再定義**だと感じています。

もちろん、GitHub Copilot CLI のように細かく制御したい場面もあります。自動化スクリプトの中で使いたいとき、細かい挙動をカスタマイズしたいとき、別の AI から呼び出したいときなどです。両者は競合するものではなく、用途に応じて使い分けるものです。

## おわりに 🚀

GitHub Copilot CLI と GitHub Copilot app は、どちらも GitHub Copilot という大きな傘の下にありながら、目指すものが根本的に異なります。

- **GitHub Copilot CLI**: 制御性の高いコマンドベース。AI が使うための CLI としても優れる。
- **GitHub Copilot app**: ヒューマンフレンドリーなエージェンティックフロントエンド。AI ネイティブな体験設計を指向。

この違いを理解することで、それぞれのツールをより意図的に使えるようになると思います。  
私自身、GitHub Copilot app を使い続けながら、AI ネイティブな開発体験がどこへ向かうのかを楽しみに見ています。✨
